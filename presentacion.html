🔑 JWT HS256 Generado en PHP · Verificado en Python

El bloque Moodle autentica al usuario y genera un token JWT. El frontend AMD envía ese token a FastAPI en cada petición. FastAPI verifica el token, gestiona la lógica y consulta al LLM local.

Flujo interno

Ciclo de vida de un mensaje

1

El alumno escribe

Pulsa Enviar en el bloque; chat.js captura el texto y la materia seleccionada.

2

Petición HTTP

El navegador envía POST /chat con el token JWT y el mensaje en el cuerpo.

3

Verificación del token

FastAPI valida la firma HS256 y extrae el user_id. Token inválido → error 401.

4

Control de límite diario

Cuenta los mensajes del usuario hoy. Si supera el límite → error 429.

5

Recuperación de contexto

Lee los últimos 10 mensajes del historial desde la base de datos.

6

Consulta al LLM

Envía system prompt + contexto + pregunta a Ollama. Si Ollama no responde → error 503.

7

Persistencia

Guarda pregunta y respuesta en la tabla block_chatbot_messages.

8

Respuesta al alumno

Devuelve un JSON; chat.js lo pinta en una burbuja en el chat.

Decisiones de diseño

¿Por qué estas tecnologías?

⚡

¿Por qué FastAPI?

Asíncrono nativo: espera al LLM sin bloquear. Validación automática con Pydantic y documentación OpenAPI generada sola.

🔒

¿Por qué LLM local?

Los datos de menores no salen del centro (RGPD). Sin coste por petición y sin dependencia de internet.

🧩

¿Por qué un bloque?

El plugin de tipo bloque se inserta en cualquier página de Moodle (curso, dashboard) sin modificar el núcleo.

🔑

¿Por qué JWT?

Stateless: el backend confía en Moodle con un secreto compartido, sin consultar la base de datos de Moodle.

🗄️

¿Por qué SQLite + MySQL?

SQLite, cero configuración para desarrollo y tests. MySQL en producción: es el motor que ya usa Moodle.

🐳

¿Por qué Docker?

Entorno reproducible e idéntico en cualquier máquina; despliegue de Moodle + base de datos en un solo comando.

Seguridad

Autenticación JWT HS256

// PHP — Moodle genera el token en el servidor private function generate_jwt(int $user_id, string $secret): string { $header = $this->b64url(json_encode(['alg'=>'HS256', 'typ'=>'JWT'])); $payload = $this->b64url(json_encode(['user_id'=>$user_id, 'exp'=>time()+3600])); $sig = $this->b64url(hash_hmac('sha256', "$header.$payload", $secret, true)); return "$header.$payload.$sig"; } # Python — FastAPI verifica el token con PyJWT def verify_jwt(token: str, secret: str = None) -> dict: return jwt.decode(token, secret, algorithms=["HS256"])

Secreto compartido

Configurado en Moodle admin panel. El mismo secreto en PHP y Python garantiza que solo Moodle puede generar tokens válidos.

Expiración 1 hora

El campo exp en el payload invalida automáticamente el token tras 60 minutos de inactividad.

user_id en payload

Cada endpoint verifica que el user_id del token coincide con el recurso solicitado. Imposible leer el historial de otro alumno.

Administración del sistema

Backend como servicio del sistema

# /etc/systemd/system/chatbot-backend.service [Unit] Description=Chatbot Académico Backend After=network.target [Service] User=www-data WorkingDirectory=/opt/chatbot/backend EnvironmentFile=/opt/chatbot/backend/.env ExecStart=/opt/chatbot/backend/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8000 Restart=always [Install] WantedBy=multi-user.target

Activar el servicio

Arranca con el servidor y se reinicia automáticamente si cae.

Ollama como servicio

El instalador oficial crea su propia unidad systemd. Solo hace falta systemctl enable ollama.

Logs del backend

journalctl -u chatbot-backend -f — seguimiento de logs en tiempo real, igual que cualquier servicio del sistema.

Proxy inverso · HTTPS

Nginx: mismo dominio, cero CORS

# /etc/nginx/sites-available/moodle server { listen 443 ssl; server_name moodle.instituto.es; ssl_certificate /etc/letsencrypt/live/moodle.instituto.es/fullchain.pem; ssl_certificate_key /etc/letsencrypt/live/moodle.instituto.es/privkey.pem; # Moodle (PHP-FPM o Apache) location / { ... } # Chatbot API — misma URL, sin CORS location /chatbot-api/ { proxy_pass http://127.0.0.1:8000/; proxy_read_timeout 120s; } }

¿Por qué el proxy inverso?

El JS del alumno corre bajo https://moodle.instituto.es. Si el backend tuviera un dominio o puerto diferente, el navegador bloquearía la petición por política CORS. Al exponerlo como /chatbot-api/ del mismo dominio, el problema desaparece.

Certificado TLS

Se genera gratis con Let's Encrypt (certbot --nginx) o se instala el certificado del propio centro educativo. El campo backend_url del plugin se fija a https://moodle.instituto.es/chatbot-api.

Retos técnicos

Dificultades encontradas

🔑

JWT cross-language

Moodle genera el token en PHP con hash_hmac y base64url manual; Python lo verifica con PyJWT. Fue necesario alinear exactamente el encoding para que las firmas coincidieran.

🤖

Parseo de JSON del LLM

Los modelos LLM no siempre devuelven JSON puro — añaden texto explicativo. Se resolvió con una expresión regular que extrae el primer objeto JSON válido de la respuesta.

🧩

Sistema AMD de Moodle

Moodle usa RequireJS (AMD) para cargar JavaScript. Fue necesario seguir la convención amd/src/ y pasar los parámetros de configuración desde PHP sin romper el ciclo de vida del módulo.

⚗️

Tests asíncronos

FastAPI usa async/await con httpx. Configurar pytest-asyncio con AsyncMock para simular Ollama sin red real requirió ajustar el event loop en cada fixture.
