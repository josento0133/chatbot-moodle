1Introducción y objetivos

Contexto

Los estudiantes de Bachillerato (16-18 años) necesitan apoyo académico más allá del horario lectivo. Las dudas surgen mientras estudian por la noche o el fin de semana, y esperar al día siguiente para preguntar al profesor rompe el ritmo de aprendizaje.

Este proyecto construye un asistente de inteligencia artificial directamente integrado en Moodle, la plataforma que los centros ya utilizan. El alumno no necesita registrarse en ningún servicio externo ni instalar nada: el chatbot aparece como un bloque más dentro de su curso habitual.

Objetivos del proyecto

Objetivos funcionales

  • Resolver dudas académicas en tiempo real mediante conversación natural
  • Generar ejercicios prácticos por materia con corrección automática
  • Registrar el progreso del alumno: consultas por materia y tasa de acierto en ejercicios
  • Mantener historial de conversación entre sesiones

Objetivos técnicos

  • Integración nativa como bloque Moodle (sin iframes ni servicios externos)
  • LLM completamente local: los datos del alumno no salen del servidor del centro
  • Autenticación segura reutilizando la sesión de Moodle
  • Suite de pruebas automatizadas con cobertura completa de la lógica de negocio

Alcance

El sistema cubre todas las materias de Bachillerato: Matemáticas, Física, Química, Biología, Lengua, Literatura, Historia, Inglés y Ciencias Sociales. Los ejercicios prácticos con corrección automática se limitan a las materias de ciencias (donde la estructura de problema-solución es más clara para el modelo).

2Arquitectura del sistema

El sistema se compone de tres capas independientes que se comunican mediante HTTP y comparten la identidad del usuario a través de tokens JWT.

→
🧩 MoodlePlugin PHP 8 + AMD JS
→
⚡ FastAPIPython 3.11 · puerto 8000
→
🤖 OllamaLlama3 · puerto 11434

Capa de presentación (Moodle)

El plugin PHP genera el HTML del bloque y un token JWT firmado con la identidad del usuario. El módulo AMD JavaScript maneja toda la interacción sin recargar la página, enviando peticiones al backend.

Capa de negocio (FastAPI)

Recibe las peticiones, verifica el token JWT, aplica el límite de mensajes diario, consulta el historial, llama al LLM y persiste los resultados en base de datos.

Capa de inteligencia (Ollama)

Servidor local que ejecuta el modelo de lenguaje. Recibe el historial de conversación y devuelve la respuesta. Completamente desconectado de internet.

Capa de datos (SQLAlchemy)

SQLite en desarrollo, MySQL/MariaDB en producción. El motor lo gestiona SQLAlchemy; el esquema lo instala automáticamente el plugin Moodle al activarse.

Flujo de una conversación

El alumno escribe un mensaje en el bloque Moodle

El módulo JavaScript recoge el texto y el token JWT que Moodle inyectó al cargar la página.

Petición POST /chat al backend

El JS envía { message, token, subject } al endpoint /chat de FastAPI.

FastAPI verifica identidad y límites

Decodifica el JWT para obtener el user_id, cuenta los mensajes del día y rechaza con 429 si supera el límite configurado.

Contexto + llamada a Ollama

Recupera los últimos 10 mensajes del historial, construye el prompt con contexto académico y llama a la API local de Ollama.

Persistencia y respuesta

Guarda el mensaje del usuario y la respuesta del asistente en base de datos. Devuelve la respuesta al navegador junto con una sugerencia de ejercicio si la materia lo permite.

5Plugin Moodle

Estructura de archivos

Clase principal (block_chatbot.php)

La clase extiende block_base y sobrescribe el método get_content(). Al renderizar el bloque:

  • Comprueba que el usuario ha iniciado sesión en Moodle (no invitado).
  • Lee la configuración (URL del backend y secreto JWT) desde los ajustes del plugin.
  • Genera un token JWT firmado con el user_id de Moodle y expiración de 1 hora.
  • Inyecta el token, la URL del backend y el ID de usuario en el módulo JavaScript.
  • Renderiza la plantilla Mustache con el HTML del chat.

Módulo AMD JavaScript (chat.js)

Utiliza el sistema de módulos AMD (RequireJS) de Moodle. El módulo exporta una función init(params) que registra todos los listeners de eventos al cargar la página.

Funciones principales

  • sendMessage() — POST a /chat
  • requestExercise() — POST a /exercise
  • submitExerciseAnswer() — POST a /exercise/answer
  • loadProgress() — GET a /progress/{userId}
  • appendMessage(role, text) — renderiza burbuja

Elementos de la interfaz

  • Área de mensajes con scroll automático
  • Indicador "..." mientras el LLM responde
  • Burbujas diferenciadas (usuario / asistente)
  • Selector de materia
  • Botón "Practicar ejercicio" (dinámico)
  • Panel de progreso desplegable

6Seguridad y autenticación JWT

El sistema no implementa su propio registro de usuarios: reutiliza completamente la sesión de Moodle. La identidad se transmite al backend mediante tokens JWT firmados con el algoritmo HMAC-SHA256 (HS256).

Flujo de autenticación

Moodle autentica al usuario (su mecanismo habitual)

El alumno inicia sesión en Moodle con su usuario y contraseña del centro. No se requiere ningún paso adicional.

PHP genera un token JWT firmado

Al renderizar el bloque, el servidor PHP crea un JWT con user_id, exp (1 hora) e iat, firmado con el secreto compartido.

El token viaja al cliente y se adjunta a cada petición

El JS lo incluye en el body o query string de cada llamada al backend. Nunca se almacena en localStorage (vive solo en memoria de la página).

FastAPI verifica el token con PyJWT

Si el token es inválido, ha expirado o la firma no coincide, la petición se rechaza con HTTP 401.

Compatibilidad PHP ↔ Python

El token se genera en PHP sin librerías externas y se verifica en Python con PyJWT. La clave es que ambos usen el mismo esquema de codificación base64url (sin padding, con sustitución de +→- y /→_): 

// PHP — implementación manual de base64url
private function b64url(string $data): string {
    return rtrim(strtr(base64_encode($data), '+/', '-_'), '=');
}

# Python — verificación con PyJWT (estándar)
def verify_jwt(token: str, secret: str) -> dict:
    return jwt.decode(token, secret, algorithms=["HS256"])

Protección de recursos

Cada endpoint que devuelve datos de un usuario comprueba que el user_id extraído del token coincide con el identificador de la URL. Así, aunque un alumno obtuviera el token de otro, no podría leer su historial ni su progreso (devuelve HTTP 403).

7Modelo de lenguaje — Ollama

Ollama es un servidor de modelos de lenguaje que corre en la misma red local que el backend. No requiere GPU (aunque mejora con una) y soporta modelos de código abierto como Llama 3 y Mistral.

Ventajas del enfoque local

  • Privacidad total: ningún mensaje del alumno sale de la red del centro
  • Sin coste por petición: no depende de APIs de pago
  • Sin límites de uso: el throughput lo decide el hardware propio
  • Actualizable: se puede cambiar el modelo con un solo comando

Modelos compatibles

  • llama3 — Meta Llama 3 8B (recomendado)
  • mistral — Mistral 7B
  • llama3:70b — versión grande (requiere GPU)
  • Cualquier modelo disponible en ollama.com

Comunicación con el backend

FastAPI llama a Ollama mediante su API REST usando httpx de forma asíncrona. El timeout es configurable (30 segundos por defecto). Si Ollama no está disponible, el backend responde con HTTP 503 y un mensaje de error claro al usuario. 

# Estructura de la petición a Ollama
{
  "model": "llama3",
  "stream": false,
  "messages": [
    { "role": "system",    "content": "<prompt del sistema>" },
    { "role": "user",      "content": "¿Cómo se resuelve una derivada?" },
    { "role": "assistant", "content": "La derivada se calcula..." },
    { "role": "user",      "content": "¿Y con la regla de la cadena?" }
  ]
}

Resultado: 28 / 28 tests superados (100%)

Los tests no dependen de red ni de Ollama instalado: se usan AsyncMock para simular el LLM y SQLite en memoria para la base de datos.

Plugin Moodle — PHPUnit

El fichero tests/phpunit/block_chatbot_test.php cubre:

  • La función b64url() produce base64url correcto (sin padding, sin + ni /)
  • El token JWT generado tiene exactamente tres partes separadas por puntos
  • El payload contiene el user_id correcto y una expiración futura
  • La firma HMAC-SHA256 es matemáticamente correcta
  • Un usuario invitado recibe el bloque vacío
  • Sin configuración del administrador, el bloque muestra un mensaje de error apropiado

Estrategia de aislamiento

Fixture autouse

El secreto JWT y el límite diario se parchean en cada test con valores conocidos. No dependen de variables de entorno.

BD efímera

Cada test crea su propia base de datos SQLite en tmp_path. Al terminar el test, se elimina. Los tests no interfieren entre sí.

Ollama simulado

AsyncMock sustituye la función call_ollama. Se puede controlar exactamente qué devuelve el modelo en cada caso.

9Manual de instalación

Requisitos previos

Software necesario

  • Docker Desktop (para el entorno de desarrollo)
  • Python 3.11 o superior
  • Ollama (ollama.com)

Recursos de hardware mínimos

  • 8 GB de RAM (16 GB recomendados)
  • 10 GB de espacio en disco (para el modelo LLM)
  • CPU multinúcleo (GPU opcional pero acelera el modelo)

Pasos de instalación

Levantar Moodle con Docker

# Desde la carpeta raíz del proyecto
docker compose up -d
# Moodle disponible en http://localhost:8080
# Admin: user=admin / password=admin123

Instalar el plugin en Moodle

Copiar la carpeta moodle-plugin/block_chatbot/ al directorio blocks/ de la instalación de Moodle, luego acceder a Administración del sitio → Notificaciones y hacer clic en Actualizar base de datos ahora.

Configurar el plugin en el panel de administración

Ir a Administración del sitio → Plugins → Bloques → Chatbot Académico y establecer:

  • URL del backend: http://localhost:8000
  • Secreto JWT: una cadena aleatoria larga (ej. 32 caracteres)

Instalar dependencias Python y arrancar el backend

cd chatbot-moodle/backend
pip install -r requirements.txt

# Crear .env con el mismo JWT_SECRET configurado en Moodle
echo "JWT_SECRET=tu-secreto-aqui" >> .env

uvicorn main:app --host 0.0.0.0 --port 8000

Descargar y arrancar el modelo LLM

ollama pull llama3       # Descarga ~4 GB, solo la primera vez
ollama serve             # Deja el servidor en ejecución

Añadir el bloque a un curso de Moodle

En cualquier curso: activar Modo de edición → Añadir un bloque → Chatbot Académico. El bloque ya aparece para todos los alumnos matriculados en ese curso.

Verificar la instalación

# Ejecutar los tests del backend
cd chatbot-moodle/backend
pytest tests/ -v
# Resultado esperado: 28 passed
